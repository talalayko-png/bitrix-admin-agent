# Архитектура

## Обзор

BSK Integration Hub — событийно-ориентированный интеграционный слой между
Bitrix24 и МойСклад. Он намеренно держит всю бизнес-логику на бэкенде, а
фронтенд (admin-панель и iframe-виджет Б24) — только UI поверх Admin API.

```
        ┌────────────┐         ┌────────────┐
        │  Bitrix24  │         │  МойСклад  │
        └─────┬──────┘         └──────┬─────┘
              │ webhook (HMAC)        │ webhook (HMAC)
              ▼                       ▼
        ┌───────────────────────────────────────┐
        │            FastAPI (app)               │
        │  /webhooks/*   /api/admin/*            │
        │                                        │
        │   WebhookService ──► WorkflowEngine    │
        │            │              │            │
        │            ▼              ▼            │
        │     OperationService (idempotency,     │
        │      approval gate, snapshots)         │
        └──────────────┬─────────────────────────┘
                       │ enqueue (Redis/RQ)
                       ▼
        ┌───────────────────────────────────────┐
        │              Worker (RQ)               │
        │  лок операции ─► workflow.execute ─►   │
        │  connectors (mock|guarded real)        │
        │  ретраи + экспоненциальный backoff     │
        └───────────────────────────────────────┘
                       │
                       ▼
            ┌──────────────────────┐
            │  БД (SQLite/Postgres)│
            │  operations, links,  │
            │  events, snapshots,  │
            │  logs, workflows     │
            └──────────────────────┘
```

## Поток обработки события

1. **Входящий вебхук** (`api/webhooks/*`) проверяет HMAC-подпись, сохраняет
   `WebhookEvent`, быстро отвечает `200 OK`.
2. **WorkflowEngine** подбирает подходящие workflow по событию и формирует
   одну или несколько **операций**.
3. **OperationService** применяет **идемпотентность** (по `idempotency_key`):
   повтор того же события не создаёт дубль. Если тип операции в списке
   `APPROVAL_REQUIRED_FOR` — операция уходит в `awaiting_approval`, иначе
   ставится в очередь.
4. **Worker** берёт задачу, захватывает **Redis-лок** по ключу операции
   (вторая попытка той же операции не исполнится параллельно), выполняет
   workflow через коннекторы и фиксирует результат/снимок.
5. При ошибке — **ретрай** с экспоненциальным backoff до `WORKER_MAX_RETRIES`,
   затем статус `failed` (можно повторить вручную из админки).

## Ключевые решения

### Предохранители вызовов
Три независимых флага (`DRY_RUN`, `USE_MOCK_CONNECTORS`, `ALLOW_REAL_API`)
гарантируют, что в MVP не происходит ни одного реального обращения к внешним
API. Любой исходящий вызов проходит через `connectors/base.py:guard_egress()`,
который бросает исключение, пока не разрешены все три флага. См. SECURITY.md.

### Очередь и абстракция backend'а
`queue/` инкапсулирует RQ. Через `QUEUE_BACKEND` выбирается:
- `redis` — реальная очередь RQ + отдельный воркер (`src/worker.py`);
- `sync` — исполнение задачи прямо в процессе (для локалки и тестов, без Redis).

Это позволяет тестам и локальной разработке работать без инфраструктуры, а
проду — использовать настоящую очередь без изменения кода.

### Идемпотентность и защита от гонок
- `idempotency_key` уникален в БД (`UNIQUE`) — двойной вебхук → одна операция.
- При исполнении воркер берёт `OPERATION_LOCK` в Redis (`SET NX PX`) по ключу
  операции — параллельное исполнение исключено даже при нескольких воркерах.

### Workflow-движок
Workflow реализуют интерфейс `Workflow` (`workflows/base.py`): `matches(event)`,
`plan(event)` → список `OperationDraft`, `execute(operation, ctx)` → результат.
Конфигурация (вкл/выкл, параметры, dry-run override) хранится в БД и
управляется через Admin API.

### Снимки (snapshots)
Перед исполнением workflow считает diff «было/стало» и сохраняет `Snapshot`.
В dry-run это и есть основной результат: что *было бы* изменено.

## Модель данных (упрощённо)

| Таблица        | Назначение                                              |
|----------------|---------------------------------------------------------|
| `operations`   | жизненный цикл операции, статус, попытки, payload/result |
| `entity_links` | связки Б24 ↔ МС (тип+id ↔ тип+id)                        |
| `webhook_events` | сырые входящие события, валидность подписи            |
| `snapshots`    | «было/стало» по операции                                |
| `operation_logs` | пошаговый журнал операции                              |
| `workflow_configs` | настройки workflow                                   |

## Технологии

- **Backend**: Python 3.11, FastAPI, SQLAlchemy 2.x, pydantic-settings, RQ,
  Redis, httpx (guarded), pytest + fakeredis.
- **Frontend**: React 18 + Vite + TypeScript (admin-panel); ванильный
  HTML/JS (bitrix24-app).
- **Инфраструктура**: docker-compose (app, worker, redis, postgres).

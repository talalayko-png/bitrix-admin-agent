```

Authorization: Bearer <ADMIN_API_TOKEN>
```

Возвращает: поля элемента (код|название|значение), определения полей,

внутри контейнера `app`; с хоста `localhost:8000` закрыт).

читаются** — их разбираем только по скриншотам владельца.

## Текущая задача: workflow «Снабжение»

Смарт-процесс Б24 «Снабжение», `entityTypeId = 1066`. Пример элемента: id 481
(https://basht.bitrix24.ru/crm/type/1066/details/481/).

Бизнес-сценарий (со слов владельца, дословно по смыслу):

На этапе **«Счёт получен / отправлен в оплату»** в МойСклад создаётся цепочка:

1. **Заказ поставщику** (`purchaseorder`):
   - галочки «проведено» и «ожидание»;
   - Склад = поле **«Склад МС»** материнской сделки;
   - План. дата приёмки = поле **«Плановая дата готовности у поставщика»**;
   - Организация, Контрагент, товары — из элемента СПА.
2. На основании заказа — **Счёт поставщика** (`invoicein`):
   - «проведено»;
   - План. дата оплаты = поле **«Дата оплаты поставщику»**;
   - Входящий номер и дата = поле **«№ и дата счёта поставщика»**.
3. На основании счёта — **Приёмка** (`supply`):
   - на дату «План. дата приёмки»;
   - галочка «Проведено» — **снята**.

Реализовано в `backend/src/workflows/create_supplier_docs.py`: цепочка из
**трёх** документов (заказ → счёт → приёмка) с маппингом реальных UF-полей,
парсером «№ и дата счёта», складом из материнской сделки и превью
`field_sources` (поле Б24 → атрибут МС). Всё в dry-run.

### Выяснено инспектором (2026-06-10, элемент 1066/481)

Реальные UF-коды СПА «Снабжение» (зашиты дефолтами в `config.py`,
переопределяются через `.env`):

| Поле Б24 (СПА)                          | Код                     | → МойСклад                        |
|-----------------------------------------|-------------------------|-----------------------------------|
| Плановая дата готовности у поставщика   | `ufCrm19_1771861585`    | `purchaseorder.deliveryPlannedMoment`, `supply.moment` |
| Дата оплаты поставщику                  | `ufCrm19_1771861774`    | `invoicein.paymentPlannedMoment`  |
| № и дата счёта поставщика (текст!)      | `ufCrm19_1771512153`    | `invoicein.incomingNumber/-Date` (пример: «1021 от 2 июня 2026 г.») |
| Ссылка на приёмку в МС (уже есть в СПА) | `ufCrm19_1772794598439` | кандидат для writeback приёмки (`BITRIX24_WRITEBACK_SUPPLY_FIELD`) |
| Материнская сделка                      | `parentId2` = 107651    | источник «Склад МС»               |
| Компания элемента (вероятно поставщик)  | `companyId` = 19771     | контрагент `agent`                |
| Покупатель - компания (отдельное поле)  | `ufCrm19_1772380474`    | не используется в закупке         |

Прочие факты (подтверждено после деплоя PR #18, 2026-06-11):
- Товарные позиции СПА **читаются** после фикса ownerType (`T` + hex,
  для 1066 — `T42a`). Позиция 481: productId 17483 «Соль таблетированная
  выварочная Экстра (25 кг)», 10 000 кг × 15 ₽ = 150 000 ₽.
- **Поставщик = companyId элемента СПА** (вопрос №2 закрыт): 19771 =
  ООО «КИРЕЕВСКИЙ СОЛЕПРОМЫСЕЛ»; компания сделки 35651 = покупатель
  (ООО «ФКПЧФ БОБИМЭКС тм»).
- **«Склад МС» сделки = `UF_CRM_1695973329`** (вопрос №4 закрыт): тип
  iblock_element (список Б24 №193), хранит **id элемента списка** (у сделки
  107651 — `15053`). Название элемента через вебхук не прочитать (нужен scope
  списков) — сопоставление через `reference-mappings` (kind=store,
  b24_value=15053 → склад МС). Код поля зашит дефолтом в config.
  Бонус: «Проект МС» = `UF_CRM_1696509078` (тоже iblock_element).
- У UF-полей **сделки** человекочитаемое имя — в `formLabel`/`listLabel`
  (а `title` = код); у полей СПА — в `title`. Инспектор это учитывает.
- «Ожидание» в заказе поставщику МС — это флаг `wait` на **позициях** заказа
  (подтверждено доками API МойСклад); workflow ставит `wait: true` всем позициям.
- Writeback-поля для id заказа/счёта в СПА пока не существуют (дефолты
  `ufCrm_PO_ID`/`ufCrm_INV_ID` — заглушки): владельцу создать поля или дать коды.
  Для приёмки есть готовое поле «Ссылка на приёмку в МС»
  (`ufCrm19_1772794598439`) — включается через `BITRIX24_WRITEBACK_SUPPLY_FIELD`.
- Включена ежечасная самопроверка прода: `.github/workflows/health-check.yml`
  (issue при недоступности /health или неожиданном выходе из dry-run).

### Что делает стороннее приложение интеграции (изучен курс htmls.ru)

Владелец пользуется приложением «Обмен данными между Битрикс24 и МойСклад»
(htmls.ru). Из курса и блога разработчика:

- Конвертация «Сделка Б24 = Заказ поставщику МС» есть (отдельная воронка
  сделок + соответствие статусов), но синхронизируются **только товары и
  статусы**; «поддержка дополнительных полей не реализована и не планируется»;
  «создание приёмки не поддерживается и не планируется» → **наш workflow не
  дублирует приложение**, а закрывает то, чего оно не умеет (даты, входящий
  №/дата, счёт поставщика, непроведённая приёмка).
- Поля сделки «Склад МС» (список №193), «Проект МС», «Идентификатор МойСклад»
  (`UF_CRM_DEAL_UID`) — созданы/используются приложением для своих
  соответствий; мы их только читаем.
- Уточнить у владельца: включена ли в приложении конвертация «Сделка = Заказ
  поставщику» для какой-то воронки (если да — следить, чтобы не было дублей
  заказов с нашим workflow).

## Текущая задача: workflow «Снабжение»

Смарт-процесс Б24 «Снабжение», `entityTypeId = 1066`. Пример элемента: id 481
(https://basht.bitrix24.ru/crm/type/1066/details/481/).

Бизнес-сценарий (со слов владельца, дословно по смыслу):

На этапе **«Счёт получен / отправлен в оплату»** в МойСклад создаётся цепочка:

1. **Заказ поставщику** (`purchaseorder`):
   - галочки «проведено» и «ожидание»;
   - Склад = поле **«Склад МС»** материнской сделки;
   - План. дата приёмки = поле **«Плановая дата готовности у поставщика»**;
   - Организация, Контрагент, товары — из элемента СПА.
2. На основании заказа — **Счёт поставщика** (`invoicein`):
   - «проведено»;
   - План. дата оплаты = поле **«Дата оплаты поставщику»**;
   - Входящий номер и дата = поле **«№ и дата счёта поставщика»**.
3. На основании счёта — **Приёмка** (`supply`):
   - на дату «План. дата приёмки»;
   - галочка «Проведено» — **снята**.

Реализовано в `backend/src/workflows/create_supplier_docs.py`: цепочка из
**трёх** документов (заказ → счёт → приёмка) с маппингом реальных UF-полей,
парсером «№ и дата счёта», складом из материнской сделки и превью
`field_sources` (поле Б24 → атрибут МС). Всё в dry-run.

### Выяснено инспектором (2026-06-10, элемент 1066/481)

Реальные UF-коды СПА «Снабжение» (зашиты дефолтами в `config.py`,
переопределяются через `.env`):

| Поле Б24 (СПА)                          | Код                     | → МойСклад                        |
|-----------------------------------------|-------------------------|-----------------------------------|
| Плановая дата готовности у поставщика   | `ufCrm19_1771861585`    | `purchaseorder.deliveryPlannedMoment`, `supply.moment` |
| Дата оплаты поставщику                  | `ufCrm19_1771861774`    | `invoicein.paymentPlannedMoment`  |
| № и дата счёта поставщика (текст!)      | `ufCrm19_1771512153`    | `invoicein.incomingNumber/-Date` (пример: «1021 от 2 июня 2026 г.») |
| Ссылка на приёмку в МС (уже есть в СПА) | `ufCrm19_1772794598439` | кандидат для writeback приёмки (`BITRIX24_WRITEBACK_SUPPLY_FIELD`) |
| Материнская сделка                      | `parentId2` = 107651    | источник «Склад МС»               |
| Компания элемента (вероятно поставщик)  | `companyId` = 19771     | контрагент `agent`                |
| Покупатель - компания (отдельное поле)  | `ufCrm19_1772380474`    | не используется в закупке         |

Прочие факты:
- `crm.item.productrow.list` вернул **ACCESS_DENIED**. Найден и исправлен баг:
  ownerType должен быть `T` + hex(entityTypeId) (для 1066 — `T42a`, мы слали
  `T1066`). Если после деплоя доступ не появится — расширить права вебхука.
- «Ожидание» в заказе поставщику МС — это флаг `wait` на **позициях** заказа
  (подтверждено доками API МойСклад); workflow ставит `wait: true` всем позициям.
- Инспектор расширен: `deal_field_overview` (поля сделки с названиями — найдём
  код «Склад МС») и `companies` (названия компаний по id — проверим поставщика).
- Writeback-поля для id заказа/счёта в СПА пока не существуют (дефолты
  `ufCrm_PO_ID`/`ufCrm_INV_ID` — заглушки): владельцу создать поля или дать коды.

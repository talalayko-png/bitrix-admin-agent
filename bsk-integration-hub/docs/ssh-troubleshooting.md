# Диагностика SSH (не подключается с компьютера)

Если SSH с вашего ПК не работает, а встроенный терминал Beget — работает,
сначала запустите серверный скрипт (он добавит запасной порт **2222**, не трогая
порт 22):

```bash
cd /opt/bitrix-admin-agent/bsk-integration-hub
git pull
bash scripts/beget-server-diagnostics.sh
```

Затем проверяйте подключение по шагам ниже.

---

## Почему ping может не проходить — и это не показатель

Многие провайдеры и хостинги (включая VPS) **режут ICMP** (ping). Отсутствие
ping **не означает**, что сервер недоступен: SSH работает по TCP (порт 22/2222),
а не по ICMP. Поэтому ориентируйтесь на `Test-NetConnection ... -Port`, а не на ping.

## Что значит `kex_exchange_identification: read: Connection timed out`

Это сообщение `ssh` означает: **TCP-соединение установилось** (порт открыт), но
во время обмена приветствиями SSH (key exchange) соединение **зависло и отвалилось
по таймауту**. То есть проблема не в том, что «порт закрыт», а в том, что **что-то
по пути рвёт уже установленную SSH-сессию**. Типичные причины:

- **VPN на роутере** (например, Keenetic) гонит трафик к серверу через туннель,
  который ломает/душит SSH;
- **DPI/ограничения провайдера** на порт 22;
- реже — проблемы MTU/фрагментации или перегрузка sshd.

Характерный признак именно вашей ситуации: `Test-NetConnection -Port 22`
**иногда** показывает доступность (TCP проходит), но `ssh` падает на хендшейке —
это почти всегда **промежуточный узел (VPN/роутер/провайдер)**, а не «сервер лежит».

## Проверка с Windows (PowerShell)

`SERVER_IP` = `155.212.188.233` (или домен `hub.bsk-grupp.ru`).

```powershell
# TCP до стандартного и запасного портов:
Test-NetConnection 155.212.188.233 -Port 22
Test-NetConnection 155.212.188.233 -Port 2222

# Подробный лог хендшейка (видно, на каком этапе зависает):
ssh -v root@155.212.188.233

# Подключение по запасному порту:
ssh -p 2222 root@155.212.188.233
```

Интерпретация:
- `2222` подключается, `22` — нет → порт 22 режется сетью/VPN/провайдером
  (см. ниже). Просто используйте `-p 2222`.
- Оба порта `TcpTestSucceeded: True`, но `ssh` виснет на `kex` → middlebox
  ломает SSH-трафик (чаще всего VPN на роутере).

## Проверка через мобильный интернет (ключевой тест)

Раздайте интернет с телефона (4G/5G) и подключитесь с ПК к этой раздаче, затем:

```powershell
ssh root@155.212.188.233
ssh -p 2222 root@155.212.188.233
```

- **Работает с телефона, но не из домашней/офисной сети** → проблема в **вашей
  локальной сети**: роутер, VPN на роутере или провайдер. Сервер исправен.
- **Не работает и с телефона** → проблема ближе к **серверу/файрволу Beget**
  (вернитесь к скрипту и панели Beget).

## Где проблема: сервер / роутер-VPN / провайдер

| Наблюдение | Вывод |
|---|---|
| С телефона (4G) SSH идёт, из вашей сети — нет | **роутер/VPN/провайдер** вашей сети |
| `:2222` идёт, `:22` нет (в любой сети) | порт **22 блокируется** сетью или провайдером |
| Оба порта TCP True, но `kex … timed out` | **VPN/DPI** ломает хендшейк (исключите IP из VPN) |
| Ни один порт не доступен и с телефона тоже | **сервер/файрвол Beget** (sshd/ufw/панель) |

## Keenetic: исключить IP сервера из VPN-маршрутизации

Если на роутере Keenetic включён VPN, который «заворачивает весь трафик»,
добавьте **статический маршрут к серверу мимо VPN** (через основного провайдера):

**Веб-интерфейс (KeeneticOS):**
1. «Интернет» → «Маршрутизация» (или «Сетевые правила» → «Маршруты») → «Добавить маршрут».
2. Тип маршрута: **маршрут до хоста**.
3. Адрес назначения хоста: `155.212.188.233`.
4. Интерфейс: ваш **основной провайдер (ISP)**, а **не** VPN-подключение.
5. Сохранить и применить.

После этого трафик к `155.212.188.233` пойдёт напрямую через провайдера, минуя VPN.

**CLI Keenetic (опционально):**
```
ip route 155.212.188.233/32 <имя_ISP_интерфейса>
system configuration save
```
(`<имя_ISP_интерфейса>` смотрите в `show interface`; обычно это основной WAN.)

**VPN-клиент на самом ПК** (не на роутере): отключите его или добавьте
split-tunnel исключение для `155.212.188.233/32`.

## Если порт 2222 работает, а 22 — нет

Это нормально и достаточно для работы — сервер слушает **оба** порта (скрипт
добавил 2222, не убирая 22). Используйте запасной порт:

```powershell
ssh -p 2222 root@155.212.188.233
```

Деплой и все команды из `docs/deploy-beget.md` при этом не меняются — они
выполняются на самом сервере. Порт 22 можно чинить отдельно (VPN/провайдер) без
спешки; на доступность сервиса и HTTPS это не влияет.

> Безопасность: скрипт не отключает порт 22, не меняет пароль root и не трогает
> Docker-проект. Чтобы убрать запасной порт позже:
> `rm /etc/ssh/sshd_config.d/99-custom-port.conf && systemctl restart ssh`.
